<task type="auto">
  <name>Task 1: Add container polling and rate limit check to instagram.ts</name>
  <files>server/instagram.ts</files>
  <action>
  Add two new exported functions to server/instagram.ts:

  1. **pollContainerStatus** — Polls container until FINISHED or ERROR:
     ```typescript
     export async function pollContainerStatus(
       containerId: string,
       accessToken: string,
       maxAttempts = 10,
       intervalMs = 15000
     ): Promise<"FINISHED" | "ERROR">
     ```
     - Loop up to maxAttempts times, waiting intervalMs between each check.
     - GET `${IG_GRAPH_BASE}/${containerId}?fields=status_code&access_token=${accessToken}`
     - If status_code is "FINISHED" -> return "FINISHED"
     - If status_code is "ERROR" or "EXPIRED" -> return "ERROR"
     - If "IN_PROGRESS" -> continue polling
     - If all attempts exhausted -> return "ERROR"
     - Use 15s intervals (not 30s) with 10 attempts = 2.5min max wait. Meta recommends "once per minute for 5 minutes" but 15s intervals give faster feedback for typical images that process in <30s.

  2. **checkRateLimit** — Queries Instagram's rate limit endpoint:
     ```typescript
     export async function checkRateLimit(
       igUserId: string,
       accessToken: string
     ): Promise<{ allowed: boolean; used: number; total: number }>
     ```
     - GET `${IG_GRAPH_BASE}/${igUserId}/content_publishing_limit?fields=quota_usage,config&access_token=${accessToken}`
     - Parse response: `data[0].quota_usage` for used, `data[0].config.quota_total` for total (default 100)
     - Return { allowed: used < total, used, total }
     - On fetch error, return { allowed: true, used: 0, total: 100 } (fail open — don't block publish on rate limit API errors)

  3. **Update postSingleImage** to use polling:
     - After creating container (existing Step 1), call `pollContainerStatus(containerId, accessToken)`
     - If result is "ERROR", return { success: false, error: "Container processing failed or timed out" }
     - Only call media_publish if pollContainerStatus returns "FINISHED"

  4. **Update postCarousel** to use polling:
     - After creating the carousel container (existing Step 2 — the parent carousel, NOT individual items), call `pollContainerStatus(carouselContainerId, accessToken)`
     - Same logic: only call media_publish if FINISHED

  Do NOT add rate limit checking inside postSingleImage/postCarousel — that belongs in the route handler (Task 2). The publish functions are pure API callers; the route orchestrates the business logic.

  2. **Add rate limit check** before attempting publish:
     - After getting credentials and before calling postToInstagram:
     ```typescript
     const rateLimit = await checkRateLimit(credentials.instagramUserId, credentials.accessToken);
     if (!rateLimit.allowed) {
       return res.status(429).json({
         error: `Instagram rate limit reached (${rateLimit.used}/${rateLimit.total} posts in 24 hours). Try again later.`
       });
     }
     ```

  3. **Set post status to "publishing"** before calling postToInstagram:
     - Update post status: `await storage.updatePost(post.id, { status: "publishing" })`
     - This prevents the scheduler from picking up the same post while container polling is in progress.

  4. **Handle publish result with error tracking:**
     - On success: update post to `{ status: "posted", publishError: null }` (clear any previous error)
     - On failure: update post to `{ status: "failed", publishError: result.error || "Unknown publish error" }`
     - Return appropriate HTTP status: 200 on success, 502 on publish failure (upstream API error)

  5. **Add try/catch around the entire publish flow** that catches unexpected errors:
     - On catch: update post to `{ status: "failed", publishError: error.message }` and return 500

  Also check if `storage.updatePost` supports the `publishError` field. If it does not (because it uses a typed update object), verify the Drizzle update accepts arbitrary column values — Drizzle's `.set()` accepts partial column objects, so `{ status: "failed", publishError: "..." }` should work if the schema has the column.
